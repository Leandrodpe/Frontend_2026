DIAGRAMA DE CLASSE:
I. Introdução à Orientação a Objetos (OO):
A Orientação a Objetos (OO) é um paradigma de programação que modela o mundo real através de objetos, que são instâncias de classes. O objetivo da OO é criar objetos que interajam entre si, encapsulando dados e comportamentos específicos.
Princípios da Orientação a Objetos
Encapsulamento: Oculta os detalhes internos de um objeto e expõe apenas o que é necessário.
Herança (Generalização/Especialização): Permite que uma classe (subclasse) herde atributos e métodos de outra classe (superclasse). Isso otimiza o tempo de desenvolvimento, diminui linhas de código e facilita manutenções.
Polimorfismo: É a capacidade de classes derivadas (subclasses) de uma mesma superclasse chamarem métodos com o mesmo nome (ou assinatura), mas que possuem comportamentos e resultados diferentes em cada subclasse.
Abstração: Representação simplificada de conceitos complexos, focando apenas nos aspectos essenciais.
II. Classes e Objetos:
Objeto Um objeto é qualquer coisa, concreta ou abstrata, que exista física ou conceitualmente no mundo real. Exemplos incluem: cliente, professor, carro. Classe Uma classe é uma abstração de um conjunto de objetos que possuem o mesmo tipo de características e comportamentos. Servem como um molde para criar objetos, especificando seus atributos e métodos.
A classe é representada por um retângulo dividido em três partes:
Nome da Classe.
Atributos (Características): Representam as características que variam de objeto para objeto (ex: nome do cliente, cor do carro) e diferenciam um objeto do outro.
Métodos (Comportamentos): São processos, serviços ou atividades que uma instância da classe pode executar.
Visibilidade (Encapsulamento):
A visibilidade é um símbolo que antecede um atributo ou método para indicar seu nível de acessibilidade:
Símbolo
+ Pública Pode ser utilizado por objetos de qualquer classe.
# Protegida Apenas objetos da classe ou de suas subclasses podem acessar.
- Privada Somente os objetos da própria classe podem utilizar.
III. Diagrama de Classes e Relacionamentos
O Diagrama de Classes tem o objetivo de permitir a visualização das classes e como elas se relacionam no sistema, fornecendo uma visão estática de sua estrutura lógica.
Relacionamento Notação e Característica
Associação Conecta duas ou mais classes, demonstrando colaboração entre as instâncias.
Associação Unária/Reflexiva Relacionamento de uma instância de uma classe com instâncias da mesma classe.
Associação Binária Conecta duas classes por meio de uma reta.
Herança (Generalização) Permite o compartilhamento de atributos e métodos entre classes. Representada por uma seta com triângulo vazio apontando para a superclasse.
Agregação Uma linha sólida com um diamante vazio na ponta da classe todo-agregado. As duas classes podem "viver" de forma independente (ligação fraca).
Composição Uma linha sólida com um diamante preenchido na ponta da classe todo-composta. Objetos da classe parte não podem viver quando o todo é destruído (ligação forte).
Classe Associativa Utilizada quando um relacionamento tem multiplicidade de "muitos" (*) em ambas as extremidades. É representada por uma reta tracejada que sai da associação e atinge a classe associativa, que guarda os objetos envolvidos
IV. Multiplicidade
A multiplicidade indica o número de objetos das classes associadas envolvidos no relacionamento.
Multiplicidade Significado:
0..1 No mínimo zero e no máximo um. Não precisa estar relacionada, mas se estiver, é com apenas uma instância.
1..1 Um e somente um. Apenas um objeto se relaciona.
0..* No mínimo nenhum e no máximo muitos. Pode ou não haver instâncias participando.
* Muitos. Muitos objetos envolvidos.
1..* No mínimo 1 e no máximo muitos. Pelo menos um objeto envolvido, podendo haver muitos.
2..5 No mínimo 2 e no máximo 5. Existe pelo menos 2 instâncias envolvidas, podendo ser 3, 4 ou 5, mas não mais do que isso.

DIAGRAMA DE CASO DE USO
I. Propósito e Conceitos Fundamentais
O Diagrama de Caso de Uso é um modelo comportamental (ou dinâmico) que mostra o que o sistema deve fazer, do ponto de vista do usuário, sem se preocupar com como o sistema fará.
Modelagem de Sistemas: É a criação de representações abstratas (diagramas) para visualizar, especificar, construir e documentar a lógica e o comportamento de um software. A modelagem visa o entendimento e a compreensão do sistema.
Requisito Funcional: Um Caso de Uso representa um requisito funcional de alto nível.
II. Elementos Chave do Diagrama
Elemento,Notação,Definição
Ator,(Boneco Palito),"É um papel desempenhado por um usuário, por outro sistema ou por um dispositivo hardware que interage com o sistema em estudo."
Caso de Uso,(Oval/Elipse),Representa uma funcionalidade ou um conjunto de ações que o sistema executa para produzir um resultado de valor para um Ator. Deve ser nomeado usando Verbo + Substantivo (ex: Realizar Inscrição).
Limite do Sistema,(Retângulo),Representa a fronteira do sistema que está sendo modelado. Os Casos de Uso ficam dentro do limite; os Atores ficam fora.
Associação,(Linha Sólida),Indica que um Ator interage (comunica) com um Caso de Uso. Essa associação sempre existe entre um ator e um caso de uso.
III. Relacionamentos entre Casos de Uso:
Os relacionamentos entre Casos de Uso permitem modularizar o sistema, evitar repetição e detalhar comportamentos.
Relacionamento,Notação,Significado,Aplicação
Inclusão,include (Seta tracejada),O Caso de Uso Base obrigatoriamente inclui e utiliza a funcionalidade do Caso de Uso Incluído. O comportamento é reutilizado por múltiplos Casos de Uso.,É usado para refatorar funcionalidades comuns. A seta aponta para o Caso de Uso incluído.
Extensão,extend (Seta tracejada),"O Caso de Uso Estendido opcionalmente adiciona funcionalidade ao Caso de Uso Base, apenas se uma condição for satisfeita.",É usado para modelar funcionalidades secundárias ou fluxos alternativos (ex: tratamento de erro). A seta aponta para o Caso de Uso Base.
Generalização,(Seta com triângulo vazio),"Um Caso de Uso (filho) herda e especializa o comportamento de outro Caso de Uso (pai), podendo sobrescrever ou adicionar novos passos.",É usado para modelar cenários onde um processo genérico é especializado (ex: Realizar Pagamento é um Caso de Uso genérico que pode ser especializado em Pagar com Cartão ou Pagar com Boleto). A seta aponta para o Caso de Uso Pai (Generalista).
IV. Descrição Formal de Caso de Uso:
Além do diagrama visual, cada Caso de Uso deve ser complementado por uma descrição textual detalhada que define o fluxo de eventos. Campo,Descrição Nome do Caso de Uso,Nome claro da funcionalidade (ex: Realizar Inscrição).
Atores,Quem interage para executar o caso de uso.
Pré-condições,O que deve ser verdadeiro antes de o caso de uso começar (ex: Aluno está identificado no sistema).
Fluxo Principal,A sequência normal e esperada de passos (passo a passo de sucesso).
Fluxo Alternativo,"Sequências de passos que podem ocorrer em condições menos comuns, mas que não são erros (ex: Aluno entra em lista de espera)."
Pós-condições,O que deve ser verdadeiro após o caso de uso ser concluído (ex: O Aluno foi inscrito em uma das turmas).
Regras de Negócio (RN),Restrições ou políticas que governam o processo (ex: RN01 – O aluno pode matricular-se no máximo em 6 disciplinas).
V. Relacionamento de Generalização de Atores
Conceito: Um Ator pode herdar as associações de outro Ator, representando uma especialização/subtipo.
Exemplo: O Ator Professor Coordenador pode ser uma especialização do Ator Professor. Ele herda todas as interações do Professor e pode ter interações adicionais.
Notação: Uma seta com triângulo vazio é desenhada do Ator Especializado para o Ator Geral.
